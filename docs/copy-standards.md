# Copy Standards

Use this project voice whenever product copy changes or new UI text is added.

## Core rules

- Lead with the user benefit, not the system behavior.
- Prefer plain, warm language over startup jargon, filler, or technical phrasing.
- Keep labels short and specific. Buttons should describe the outcome.
- Avoid AI-generic phrasing like "streamline," "seamless," "unlock," "optimize," or "enhance."
- Avoid devy wording like "submit," "execute," "invoke," "payload," "sync issue," or "validation error" unless the user truly needs technical detail.
- Error states should explain what happened, what the user can do next, and whether their work is safe.
- Empty states should invite the next useful action instead of stating the obvious.
- If the app is waiting, say what is happening in human terms.

## Preferred tone

- Calm
- Clear
- Direct
- Slightly warm
- Never cute at the expense of clarity

## Examples

- Prefer "Invite members" over "Manage collaborators"
- Prefer "Try again in a minute" over "Rate limit exceeded"
- Prefer "Your access to this trip was removed" over "Authorization revoked"
- Prefer "Email is not available right now. Copy the link instead." over "Email provider unavailable"
- Prefer "Set dates to frame the itinerary" over "No travel window configured"

## Error-writing checklist

- Does it explain the problem without blaming the user?
- Does it avoid internal system language?
- Does it offer a next step when one exists?
- Would it still make sense if a non-technical traveler read it?
